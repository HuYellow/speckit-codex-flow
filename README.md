# Spec Kit Codex Flow

[中文说明](./README.zh-CN.md)

Spec Kit Codex Flow is a local Codex plugin that brings [GitHub Spec Kit](https://github.com/github/spec-kit) into a stricter Codex development loop. It keeps the official Spec Kit workflow as the source of truth, then adds Codex-native skills and MCP tools for bootstrapping, status inspection, and implementation gates.

The plugin is designed for local personal use first. It helps Codex answer three practical questions before writing code:

- Is this repository initialized for Spec Kit with the official Codex skills integration?
- Which Spec Kit phase should happen next?
- Is implementation allowed, or are the spec artifacts still incomplete?

## Why This Exists

Spec Kit promotes spec-driven development: define product intent, clarify ambiguity, create a technical plan, break it into tasks, then implement from that plan. In normal day-to-day Codex usage, it is easy to skip directly from a vague feature request into code changes.

This plugin adds a lightweight workflow layer that makes the intended path explicit:

```text
constitution -> specify -> clarify -> plan -> analyze -> tasks -> implement -> converge
```

It does not replace Spec Kit and does not copy Spec Kit templates by hand. Project-local official skills such as `$speckit-specify`, `$speckit-plan`, and `$speckit-implement` are still generated by the official `specify` CLI. This plugin only orchestrates those skills and checks whether the artifacts are ready.

## Core Principle

The plugin separates responsibilities:

- Official Spec Kit owns templates, scripts, and project-local `$speckit-*` skills.
- Codex skills in this plugin guide the user through the workflow and explain blockers.
- MCP tools inspect local filesystem state and run explicit setup commands.
- Strict gates prevent this plugin's implementation entrypoint from writing code when required artifacts are missing or unresolved.

This keeps the workflow predictable without forking or reimplementing GitHub Spec Kit.

## Architecture

```text
Codex chat
  |
  |-- plugin skills
  |     |-- $speckit-flow
  |     |-- $speckit-bootstrap
  |     |-- $speckit-specify
  |     |-- $speckit-status
  |     |-- $speckit-gate
  |     `-- $speckit-flow-implement
  |
  `-- MCP server
        |-- check_environment
        |-- install_specify_cli
        |-- bootstrap_project
        |-- project_status
        `-- gate_phase

Target project
  |
  |-- .specify/
  |-- .agents/skills/speckit-*/
  `-- specs/<feature>/{spec.md,plan.md,tasks.md}
```

## Included Codex Skills

| Skill | Purpose |
| --- | --- |
| `$speckit-flow` | Top-level workflow controller. It reads project state and recommends the next Spec Kit skill. |
| `$speckit-bootstrap` | Checks the local environment, installs or guides installation of `specify`, and initializes a project with Codex skills mode. |
| `$speckit-specify` | Plugin-level compatibility trigger for creating or updating a feature spec. It delegates to the project-local official `speckit-specify` instructions. |
| `$speckit-status` | Read-only project status summary for `.specify`, official skills, and current feature artifacts. |
| `$speckit-gate` | Read-only phase gate for `plan`, `tasks`, `implement`, and `converge`. |
| `$speckit-flow-implement` | Strict implementation entrypoint. It only proceeds to official `$speckit-implement` after the implementation gate passes. |

## MCP Tools

| Tool | Behavior |
| --- | --- |
| `check_environment` | Detects `codex`, `uv`, `specify`, PowerShell, Git root, and pinned Spec Kit metadata. |
| `install_specify_cli` | Installs `specify-cli` through `uv`, pinned by default to `v0.12.2`. If `uv` is missing, it returns installation guidance instead of failing silently. |
| `bootstrap_project` | Runs official `specify init ... --integration codex --integration-options=--skills` for a new or existing project. |
| `project_status` | Reads `.specify/memory/constitution.md`, `.agents/skills/speckit-*`, and `specs/*/{spec.md,plan.md,tasks.md}`. |
| `gate_phase` | Applies strict workflow checks before `plan`, `tasks`, `implement`, or `converge`. |

## Gate Rules

The strict implementation gate blocks when:

- The project is not initialized with Spec Kit.
- `.specify/memory/constitution.md` is missing.
- Official project-local Codex skills under `.agents/skills/speckit-*` are missing.
- No current feature exists under `specs/`.
- `spec.md`, `plan.md`, or `tasks.md` is missing.
- Spec artifacts still contain `[NEEDS CLARIFICATION]` or unresolved template placeholders.
- `tasks.md` has no executable checkbox tasks.
- The constitution appears to require tests, but `tasks.md` has no test tasks.

The plugin does not prevent you from using Codex outside this flow. It only enforces gates when you use this plugin's workflow skills.

## Defaults

- Spec Kit version: `v0.12.2`
- Spec Kit release date: `2026-06-30`
- Spec Kit repository: `https://github.com/github/spec-kit`
- Spec Kit integration: `codex`
- Integration mode: `--skills`

## Installation

Clone the plugin into your local plugin directory:

```powershell
git clone https://github.com/HuYellow/speckit-codex-flow.git C:\Users\$env:USERNAME\plugins\speckit-codex-flow
cd C:\Users\$env:USERNAME\plugins\speckit-codex-flow
npm install
npm run build
npm test
```

Add it to your personal marketplace if it is not already present:

```json
{
  "name": "speckit-codex-flow",
  "source": {
    "source": "local",
    "path": "./plugins/speckit-codex-flow"
  },
  "policy": {
    "installation": "AVAILABLE",
    "authentication": "ON_INSTALL"
  },
  "category": "Productivity"
}
```

Then install it into Codex:

```powershell
codex plugin add speckit-codex-flow@personal
```

Start a new Codex thread after installation so Codex loads the plugin's skills and MCP tools.

## Usage

### 1. Check Status

Ask Codex:

```text
$speckit-status
```

This reports whether the current repository has Spec Kit initialized, which official skills exist, which feature is current, and which artifact is blocking progress.

### 2. Bootstrap a Project

In an existing repository:

```text
$speckit-bootstrap
```

The plugin checks for `uv` and `specify`, installs `specify-cli` when possible, then runs the official Codex skills initialization:

```powershell
specify init . --integration codex --integration-options=--skills
```

For a new project, provide the target path in your prompt and the plugin will use `mode: "new"`.

### 3. Follow the Workflow

Use:

```text
$speckit-flow
```

The plugin reads project state and recommends the next official Spec Kit skill, such as `$speckit-constitution`, `$speckit-specify`, `$speckit-plan`, or `$speckit-tasks`.

### 4. Create a Feature Spec Directly

You can also call the specification phase directly:

```text
$speckit-specify Build a local dashboard for tracking project risks and mitigation owners.
```

This wrapper checks that the project has been bootstrapped, then reads and follows the official project-local `.agents/skills/speckit-specify/SKILL.md` instructions.

### 5. Gate Implementation

Before implementation:

```text
$speckit-flow-implement
```

If the gate passes, Codex can continue with official `$speckit-implement`. If it fails, the plugin reports concrete blockers and the next skill to run.

### 6. Reconcile Drift

After implementation, use:

```text
$speckit-gate converge
```

Then run official `$speckit-converge` when appropriate to compare the codebase against the spec, plan, and tasks.

## Troubleshooting

### `uv` is missing

Install `uv` first, then run `$speckit-bootstrap` again.

Reference: `https://docs.astral.sh/uv/getting-started/installation/`

### `specify` is missing

Run `$speckit-bootstrap`. If `uv` is available, the plugin can install `specify-cli` pinned to `v0.12.2`.

### Skills do not appear in Codex

Reinstall the plugin:

```powershell
codex plugin add speckit-codex-flow@personal
```

Then start a new Codex thread.

### The implementation gate blocks

Run `$speckit-status` or `$speckit-gate implement`. The blocker text is intentional: fix the missing or unresolved artifact, then re-run `$speckit-flow-implement`.
